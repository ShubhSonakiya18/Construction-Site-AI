# Sample Audio — Speech Processing Framework

This directory holds audio fixtures used by the Sprint 3 Speech Processing
Framework (`speech/`) for development, validation testing, and (eventually)
transcription quality benchmarking.

## What's here now

10 synthetic sine-tone WAV files, generated by `scripts/create_sample_audio.py`:

| File | Duration | Frequency | Sample Rate | Channels | Purpose |
|---|---|---|---|---|---|
| `sample_01_short_tone.wav` | 1.0s | 440 Hz | 16000 Hz | mono | Minimal valid duration |
| `sample_02_medium_tone.wav` | 5.0s | 440 Hz | 16000 Hz | mono | Standard case |
| `sample_03_long_tone.wav` | 15.0s | 440 Hz | 16000 Hz | mono | Multi-chunk boundary |
| `sample_04_low_freq.wav` | 5.0s | 220 Hz | 16000 Hz | mono | Frequency variation |

Each `.wav` has a matching `.txt` file containing a placeholder ground-truth
string. **These are not real transcripts** — sine tones have no speech content.

## Why synthetic audio

Real foreman voice recordings cannot be committed to this repository:
- They may contain identifiable voices or site-specific information
- Binary audio files bloat git history
- No consistent, license-clean source of construction-site speech exists

Synthetic sine-tone WAVs exercise every code path that doesn't require actual
speech recognition: format detection, duration limits, sample rate checks,
channel handling, chunking math, and the full pipeline orchestration
(validation → preprocessing → STT call → postprocessing → result).

## Regenerating

```bash
python scripts/create_sample_audio.py
python scripts/create_sample_audio.py --output-dir data/sample_audio
```

Requires `numpy` and `soundfile` for proper sine-tone generation. Without
them, the script falls back to silent WAV files (still valid containers, but
useless for anything beyond format/structure testing).

## Adding real audio for WER (Word Error Rate) testing

To test actual transcription accuracy:

1. Record or obtain a real WAV/MP3 file with clear speech (construction
   terminology ideally — "check the rebar on level two," etc.)
2. Save it as `data/sample_audio/<your_name>.wav`
3. Create `data/sample_audio/<your_name>.txt` with the exact ground-truth
   transcript
4. Run the pipeline and compare output against the ground truth using `jiwer`

Real audio files are gitignored by default (see repository `.gitignore`) —
add an explicit exception if you want to commit a specific clean sample.

The acceptance tests in `tests/test_audio_pipeline.py` that depend on real
audio are marked `@pytest.mark.skipif` and only run when real WAV files are
present in this directory.
