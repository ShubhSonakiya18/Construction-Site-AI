# datasets/

Synthetic construction site data generated by the **Construction Site AI Data Generation Framework** (Sprint 2).

## Directory Structure

```
datasets/
├── raw/          Intermediate records before validation (staging area)
├── generated/    All generated records, including those with warnings
├── validated/    Records that passed all 4 validation phases
└── exports/      Final consumer-ready datasets in their target formats
```

## Datasets

### daily_logs_v1.jsonl — 5,000 records (JSONL)

**What:** One construction daily log per working day per simulated project.
Generated by simulating ~50 residential single-family construction projects
from site preparation through project closeout.

**Schema:** ConstructionDailyLog v1.0.0 (see `knowledge/construction_daily_log_schema.json`)

**Key fields:**
- `current_stage` — Active construction stage (22 valid enum values)
- `weather.*` — Daily weather conditions and productivity impact
- `workforce.*` — Trades on site, worker counts, late arrivals
- `work_completed[]` — Tasks completed today
- `materials.*` — Materials used and delivered
- `safety.*` — PPE compliance, incidents, safety meetings
- `delays[]` — Delay events with root cause
- `inspections[]` — Municipal inspections conducted
- `financials.*` — Daily cost breakdown

**Training target:** This is the primary dataset for AI model training in Sprint 3+.

---

### project_schedules_v1.jsonl — 1,000 records (JSONL)

**What:** One record per simulated project showing planned vs. actual schedule.
Includes per-stage planned/actual start/end dates and delay breakdown by category.

**Key fields:**
- `schedule_status` — `on_time | delayed | ahead_of_schedule`
- `stage_schedules[]` — Per-stage planned vs. actual dates
- `delay_breakdown_by_category[]` — Weather/material/labor delay attribution
- `total_delay_days` — Total schedule slip in calendar days

---

### customer_updates_v1.jsonl — 1,000 records (JSONL)

**What:** Training pairs for the customer update generation AI model.
Each record contains a realistic foreman's terse field note and the professionally
written customer email that should be produced from it.

**Key fields:**
- `raw_foreman_notes` — Terse, technical, jargon-filled field notes
- `customer_email_subject` — Appropriate subject line
- `customer_email_body` — Professional customer-facing update
- `stage_context` — Which construction stage the note is from
- `expansion_ratio` — How much longer the email is vs. the raw note

**Training target:** Fine-tuning an LLM to translate construction jargon to client-appropriate language.

---

### safety_talks_v1.csv — 200 records (CSV)

**What:** Toolbox talk / safety briefing records from construction sites.
Based on OSHA 29 CFR 1926 standards and hazards from the construction ontology.

**Key columns:**
- `talk_topic` — Safety topic(s) discussed
- `osha_reference` — Applicable OSHA regulation
- `hazards_discussed` — Hazards identified (from ontology)
- `ppe_discussed` — PPE types covered
- `attendees_count` — Number of workers who attended
- `stage_context` — Construction stage the talk was related to

---

### materials_v1.csv — 500 records (CSV)

**What:** Construction material catalog entries.
Material names and categories are sourced from `construction_ontology.json`.

**Key columns:**
- `material_name` — Material name with size/grade specification
- `category` — Material category (concrete, lumber, electrical, etc.)
- `unit_of_measure` — Unit (cubic_yard, sheet, linear_feet, etc.)
- `unit_price_usd` — Typical unit price
- `primary_stage_used` — Stage where this material is primarily used
- `applicable_stages` — All stages where this material may appear

## Generation

To regenerate all datasets:

```bash
python generate.py
```

To regenerate a specific dataset:

```bash
python generate.py --dataset daily_logs --count 5000 --seed 42
```

To scale to 500,000 records, change only the count constants in `dataset_generation_framework/config.py`:

```python
DAILY_LOG_COUNT = 500_000   # was 5_000
```

## Validation

All records are validated through a 4-phase pipeline before being written to exports/:

| Phase | Severity | Behavior |
|-------|----------|----------|
| Phase 1 | Blocking | Record not written; generation continues |
| Phase 2 | Error | Record stored as draft; must be reviewed |
| Phase 3 | Warning | Record stored; flagged as unusual |
| Phase 4 | Info | Advisory metadata only |

See `dataset_generation_framework/validation/pipeline.py` for details.

## Statistics

After generation, a report is written to `datasets/exports/generation_report.json`
with record counts, generator performance metrics, and content distribution analysis.
